# User-Directed Agent Workflow

> **Status:** Active direction, reflected in worker behavior on 2026-05-05.

## Purpose

WekiFlow is a knowledge-management product, not only an autonomous ingestion pipeline. The agent must help users turn knowledge inputs into maintained Markdown wiki pages.

That includes:

- company policies and internal rules
- personal notes
- meeting notes and action items
- user-provided data documents
- external AI/webhook/API payloads
- existing folders and pages selected as source material

The agent can still run scheduled maintenance, but user-directed document work is a first-class workflow.

## Operating Modes

### External ingestion

External signals arrive through ingestion APIs or webhooks. The agent classifies and applies them through the normal confidence gates.

### User-directed wiki edit

A user selects pages, folders, or pasted material and gives an instruction such as:

- write a new document from these notes
- merge these policy fragments into a canonical page
- summarize these meeting notes and create an action-item page
- edit this page to reflect the attached data
- append this regulation update to the relevant section

In this mode, the user instruction is the primary task. The agent must not reinterpret the request as generic cleanup unless the user asked for cleanup.

### Scheduled maintenance

Scheduled tasks reuse the same agent loop, but the instruction may be generated by a cron task or workspace policy. Maintenance can include stale-page cleanup, dedupe, restructuring, or other recurring wiki hygiene.

## Behavioral Contract

- Markdown remains the source of truth.
- Every AI mutation creates a revision and preserves provenance through `agent_runs`, `model_runs`, `ingestions`, `ingestion_decisions`, and `audit_logs`.
- Selected pages may be source material, edit targets, or both. Their role is inferred from the user instruction.
- When selected pages are source material for a new document, the agent should synthesize a complete Markdown page rather than route the work to cleanup or noop.
- For small selected scopes, the worker prefetches selected pages with `read_page(format="markdown")` before planning so the model has exact source content.
- The agent must not claim "insufficient context" for selected pages until it has attempted to read the needed source content.
- When a user explicitly asks for a new page, `create_page` is valid after duplicate checks; the request should not be downgraded merely because existing pages could also hold the content.
- Selected source pages are preserved unless the user explicitly asks to delete, archive, or destructively merge them.
- `request_human_review` is used when the user asked for a change but the agent cannot execute it safely.
- `noop` is used only when no wiki change is needed.

## Implementation Notes

- `packages/worker/src/lib/scheduled/input-adapter.ts` frames selected-page runs as a "User-directed wiki edit request".
- `packages/worker/src/lib/agent/loop.ts` prepends scheduled user-directed instructions to the plan prompt.
- `packages/worker/src/lib/agent/loop.ts` prefetches selected seed pages before planning, defaulting to 20 pages via `AGENT_SCHEDULED_SEED_READ_LIMIT`.
- New pages requested from a folder-scoped run carry `targetFolderId` through API, job payload, internal ingestion, and agent state so `create_page` lands in the intended folder.

## Product Direction

Future agent work should be evaluated by whether it improves the user's ability to manage knowledge:

- turn raw source material into canonical Markdown pages
- keep originals traceable while producing synthesized documents
- edit and merge existing wiki pages without losing provenance
- preserve human supervision for ambiguous or risky operations
- expose enough trace context that users understand what the agent read, planned, and changed
