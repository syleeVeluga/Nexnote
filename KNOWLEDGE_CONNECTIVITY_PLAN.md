# 지식 연결성 구현 계획

## 문서 목적

이 문서는 NexNote의 지식 연결성을 과도한 재설계 없이 강화하기 위한 구현 기준 문서다.

이번 계획의 초점은 다음 3가지다.

- 페이지 부모-자식 구조가 탐색 경험에 실제로 반영될 것
- 문서 간 명시 링크와 의미 그래프를 분리하되 함께 보이게 할 것
- 기존 데이터 모델과 API를 가능한 한 유지할 것

이 문서는 다음 커밋의 기준점으로 사용하고, 이후 작업은 이 문서를 기준으로 진행한다.

## 현재 상태 요약

현재 시스템은 연결이 두 축으로 나뉘어 있다.

- 구조 연결: `pages.parentPageId`
- 의미 연결: `entities` / `triples`

이 분리는 설계상 맞지만, 사용자 입장에서는 같은 지식 묶음이 분리되어 보이는 문제가 있다.

대표적인 현상은 다음과 같다.

- B를 A의 하위 페이지로 넣어도 A의 그래프에서 B의 지식이 보이지 않는다.
- 하위 페이지는 트리에서는 연결되지만 그래프에서는 연결되지 않은 것처럼 보인다.
- 문서 간 내부 링크와 백링크가 아직 없다.

## 설계 원칙

### 1. 연결은 3종류로 분리한다

#### 1-1. 구조 연결

- 부모-자식 페이지 관계
- 문서 정리 구조를 나타냄
- 진실 원천은 `pages.parentPageId`

#### 1-2. 문서 연결

- 페이지 링크, 멘션, 백링크
- 문서가 다른 문서를 명시적으로 참조하는 관계

#### 1-3. 의미 연결

- `entities` / `triples` 기반 지식 그래프
- 문서 내용에서 추출된 사실 관계

### 2. 이번 범위에서 하지 않을 것

- 부모-자식 관계를 자동으로 triple로 저장
- `parent_of` 같은 관계를 의미 그래프에 강제 주입
- Graph DB 도입
- 처음부터 전체 워크스페이스 그래프를 메인 UX로 재설계
- 구조 연결, 문서 연결, 의미 연결을 하나의 edge 모델로 통합

## 목표 상태

사용자는 다음을 자연스럽게 경험해야 한다.

- Sidebar에서는 페이지 구조가 보인다.
- 문서에서는 어떤 페이지를 참조하는지와 누가 이 문서를 참조하는지가 보인다.
- Graph에서는 현재 페이지만 볼 수도 있고, 하위 페이지까지 포함한 지식 묶음을 볼 수도 있다.

핵심은 “연결 모델을 하나로 합치는 것”이 아니라 “연결된 경험을 만드는 것”이다.

## 구현 방향

## Phase 1. subtree 그래프 범위 추가

### 핵심 아이디어

기존 그래프 조회는 유지하고, 조회 범위만 확장한다.

- `scope=self`: 현재 페이지만 기준으로 그래프 조회
- `scope=subtree`: 현재 페이지와 모든 하위 페이지를 포함해 그래프 조회

기본값은 `self`로 유지한다.

### 이유

- 기존 동작과 호환된다.
- 현재 그래프 API를 폐기하지 않고 확장만 하면 된다.
- 페이지 이동이 구조 연결에만 영향을 주고, triple 데이터 자체를 오염시키지 않는다.
- 사용자는 부모 페이지에서 하위 지식 묶음을 한 번에 탐색할 수 있다.

### 구현 포인트

- `graphQuerySchema`에 `scope` 추가
- 허용값: `self | subtree`
- API에서 `sourcePageId = pageId` 단일 조건을 `pageId 집합` 조건으로 확장
- `subtree`일 때 재귀 쿼리로 하위 페이지 ID 집합 계산
- 기존 응답 포맷은 유지하고 `meta.scope` 정도만 선택적으로 추가

### 예상 변경 파일

- `packages/shared/src/schemas/page.ts`
- `packages/api/src/routes/v1/pages.ts`
- `packages/web/src/lib/api-client.ts`
- `packages/web/src/components/graph/GraphPanel.tsx`

### 완료 조건

- `self` 조회 결과는 기존과 동일하다.
- `subtree` 조회 시 현재 페이지와 하위 페이지의 트리플이 함께 반영된다.
- 하위 페이지가 없으면 `subtree` 결과는 사실상 `self`와 동일하다.
- 부모 변경 후 `subtree` 그래프 결과가 새 구조를 반영한다.

## Phase 2. 문서 링크와 백링크 추가

### 핵심 아이디어

하위 페이지 구조만으로는 위키 연결성이 충분하지 않다. 문서가 다른 문서를 참조하는 연결이 별도로 필요하다.

### 방향

- 문서 내 페이지 링크 또는 멘션 문법 추가
- 저장 시 링크를 추출해서 `page_links` 같은 별도 엣지로 저장
- 페이지 하단에 backlinks 패널 제공

### 이유

- 사용자가 “왜 연결되는가”를 이해할 수 있다.
- 그래프와 별개로 위키다운 연결성을 제공할 수 있다.
- 향후 관련 페이지 추천과 링크 탐색의 기반이 된다.

### 예상 변경 범위

- 저장 시 링크 추출 로직
- 링크 저장 테이블 또는 동등한 저장 모델
- backlinks 조회 API
- 페이지 상세 화면의 backlinks UI

### 완료 조건

- 문서 내 페이지 링크를 저장하고 조회할 수 있다.
- 특정 페이지가 어떤 페이지들로부터 참조되는지 확인할 수 있다.
- 구조 연결과 문서 연결이 서로 분리된 상태로 노출된다.

## Phase 3. UI 정리

### UI 원칙

- Sidebar / Tree: 구조 연결
- Backlinks / Related Pages: 문서 연결
- Graph Panel: 의미 연결

Graph Panel 안에서는 범위만 토글한다.

- 현재 페이지만 보기
- 하위 페이지 포함 보기

### 완료 조건

- 사용자가 현재 보고 있는 연결 종류를 혼동하지 않는다.
- 구조 연결과 의미 연결이 한 화면에서 뒤섞여 보이지 않는다.

## 기술 결정

### 유지할 것

- PostgreSQL 기반 `triples` 모델
- `pages.parentPageId` 기반 계층 구조
- 기존 per-page 그래프 API의 기본 동작

### 추가할 것

- 그래프 조회 범위 개념 `scope`
- subtree page set 계산
- 문서 링크 및 backlinks

### 배제할 것

- 부모-자식 관계의 triple 자동 생성
- 구조 관계를 의미 관계로 간주하는 설계
- 별도 그래프 스토리지 도입

## 테스트 기준

### Phase 1

- `scope=self`는 기존 결과와 동일해야 한다.
- `scope=subtree`는 하위 페이지의 엔티티와 엣지를 포함해야 한다.
- 부모 재배치 후 `subtree` 조회 결과가 변경되어야 한다.
- 하위 페이지가 매우 많을 때도 기존 `limit`와 보호 로직이 깨지지 않아야 한다.

### Phase 2

- 문서 저장 시 링크가 정상 추출된다.
- 링크 삭제 시 backlinks에서도 제거된다.
- 잘못된 링크는 안전하게 무시되거나 검증 오류로 처리된다.

## 최종 정리

이번 구현의 핵심은 연결을 무작정 늘리는 것이 아니라, 연결의 종류를 분리한 채 사용자가 연결된 지식으로 탐색하게 만드는 것이다.

최소 구현으로 적절한 방향은 다음과 같다.

- 구조 연결은 그대로 유지한다.
- 의미 그래프는 subtree 범위 조회를 추가한다.
- 문서 링크와 백링크는 별도 계층으로 추가한다.
- UI에서는 세 연결을 섞지 않고 역할별로 보여준다.

이 방향이면 프로젝트 목적에 맞고, 특히 하위 페이지 연결성을 자연스럽게 반영하면서도 오버 스펙으로 흐르지 않는다.
